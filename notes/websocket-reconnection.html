Designing for Reconnection: WebSocket State at Scale

In progress — March 2026

Most WebSocket tutorials end at "connection established." The real engineering problem begins when that connection drops — and at scale, it will drop constantly. Mobile clients switch networks. Load balancers time out idle connections. Servers restart. The design question is not how you handle a connected client; it is what state the client is in when it comes back.

The state problem

A WebSocket connection is stateful by nature. The challenge is that your application state does not have to be. In the ATalk messaging platform, the connection layer held no application state — session data lived in Redis, message queues lived in RabbitMQ. When a client reconnected to a different server instance, the new instance fetched session state from Redis and replayed any unacknowledged messages from the RabbitMQ queue. The connection was ephemeral; the data was not.

Message replay vs. polling

The naive approach to reconnection is to have the client re-fetch state from a REST endpoint. This works at low scale but creates a polling spike whenever a large number of clients reconnect simultaneously — the thundering herd problem. A better model is cursor-based replay: the client sends its last acknowledged message ID on reconnect, and the server streams only the delta. This is how Kafka consumers work, and it is the right model for any system where message ordering matters.

Thundering herd on reconnect

When a server restarts, all connected clients attempt to reconnect within a short window. Naive retry logic (reconnect immediately on disconnect) converts a single server restart into a thundering herd. The mitigation is exponential backoff with jitter on the client side, combined with a connection rate limiter on the server side. Socket.io supports configurable reconnection backoff; raw WebSocket implementations need this added explicitly.

Sticky sessions vs. session-agnostic design

Sticky sessions (routing a client to the same server instance on reconnect) are tempting but fragile — they fail when the server instance is gone, which is exactly when reconnection happens. The better design makes the connection layer session-agnostic: any instance can handle any client, because session state is external. This requires more infrastructure (Redis for session, a message bus for delivery) but eliminates a whole class of failure modes.
