Kafka vs RabbitMQ: A Systems Engineering Decision

In progress — March 2026

This is not a feature comparison. Both tools work. The question is which mental model matches the problem you are solving. Choosing the wrong one creates friction at every layer — the API fights you, the operational model fights you, and the failure modes surprise you.

Two different models

RabbitMQ models the routing problem. A producer publishes a message to an exchange; the exchange routes it to one or more queues based on bindings; consumers read from queues. Once a consumer acknowledges a message, RabbitMQ deletes it. The broker is smart; the consumers are simple.

Kafka models the log problem. A producer appends events to a topic partition. Consumers read from an offset and track their own position. Messages are retained for a configurable period — independent of whether they have been consumed. The log is durable; consumers can replay it. The broker is simple; the consumers track their own state.

The decision framework

Question Points to RabbitMQ Points to Kafka
Do messages have individual routing logic? Yes — per-message routing, topic exchanges, fanout No — all consumers in a group get the same partition
Do consumers need to replay historical events? No — message deleted on ack Yes — offset reset to any point in the log
Is delivery acknowledgement per-message critical? Yes — consumer ack model is first-class Offset-based; partial ack is awkward
Do you need event sourcing or audit trail? Not naturally Yes — the log is the source of truth
Is throughput the primary constraint? Moderate throughput; operational simplicity High throughput; partition-based parallelism

How this played out in practice

ATalk (banking messaging platform) used RabbitMQ. The problem was per-user message routing with guaranteed delivery — each user had a durable queue, messages were routed to the right queue by the exchange, and consumer acknowledgement ensured no message was lost on disconnect. RabbitMQ's model matched exactly.

An IoT analytics pipeline with replay requirements — where a new downstream consumer needs to process all historical telemetry, not just messages going forward — is a Kafka problem. The log retention model makes this trivial. Trying to solve it with RabbitMQ requires a separate persistence layer and complicates the consumer significantly.
